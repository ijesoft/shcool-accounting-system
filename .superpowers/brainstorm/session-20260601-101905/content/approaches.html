Choose the Architectural Approach

School Accounting System — Mixed K-12 + College, Multi-Entity, PH GAAP, 100 concurrent users, 100k transactions/month, On-Premise

A

Modular Monolith + Event-Driven Posting (Recommended)

Pros

  • Single deployable — simple ops for on-premise
  • Domain modules with clear boundaries (DDD)
  • Event-driven posting engine with outbox pattern
  • Multi-entity via tenant_id at schema level
  • ACID compliance within single database
  • Easy to monitor, backup, and restore
  • Best fit for 100 users / 100k txns per month

Cons

  • Cannot scale modules independently
  • Database becomes single point of scaling
B

Microservices + CQRS / Event Sourcing

Pros

  • Independent scaling per domain
  • Event sourcing provides native audit trail
  • Separate read models for reporting
  • Team scalability (multiple dev teams)

Cons

  • Significant ops complexity for on-premise
  • Distributed transactions for accounting are dangerous
  • Eventual consistency risks for financial data
  • Overkill for 100 users / 100k txns
  • Harder to backup and restore consistently
C

Modular Monolith + Database-Centric Posting Engine

Pros

  • PostgreSQL stored procedures enforce accounting rules at DB level
  • Highest performance for journal posting
  • ACID guarantees at database level
  • Impossible to post unbalanced entries

Cons

  • Business logic trapped in stored procedures — hard to test, version, debug
  • Poor developer experience with PL/pgSQL
  • Makes Next.js a thin UI layer only
  • Vendor lock-in to PostgreSQL
  • Cannot reuse logic across services if migrated later

My recommendation: Approach A — modular monolith with event-driven posting. It balances clean architecture with operational simplicity for on-premise deployment. The event-driven approach gives us an audit trail without the complexity of full event sourcing.
